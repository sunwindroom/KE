## 2. 架构设计目标与原则

### 2.1 设计目标

- **知识驱动**：以PHM领域知识图谱与知识库为核心资产，模型与应用围绕知识资产构建；
- **模型可插拔**：大模型层与上层应用解耦，支持模型替换、多模型并存、领域微调模型灵活切换；
- **安全可控**：满足私有化部署、数据不出域、分级分类访问控制要求；
- **弹性扩展**：支持知识规模、用户并发、业务领域的水平扩展；
- **人机协同闭环**：架构须支撑"AI初步生成—专家审核确认—知识入库—反哺模型"的闭环机制。

### 2.2 架构设计原则

| 原则 | 说明 |
|---|---|
| 分层解耦 | 数据接入、知识加工、知识存储、模型推理、应用交互各层职责单一，通过标准接口交互 |
| 微服务化 | 核心能力（抽取、检索、问答、图谱查询、Agent编排）以微服务形式独立部署、独立扩展 |
| 领域驱动设计（DDD） | 按能源/交通/航空航天领域划分知识空间（Bounded Context），核心通用能力下沉共享 |
| 异步与流式处理 | 知识抽取、批量入库、模型训练等重任务采用异步队列处理，问答/检索采用流式响应提升体验 |
| 安全默认（Security by Default） | 所有对外/对内接口默认鉴权，默认最小权限，密级过滤前置于检索环节 |

| 层级 | 核心职责 |
|---|---|
| 数据接入层 | 完成多源异构数据的采集、格式转换、初步校验，统一输出至知识加工层 |
| 知识加工层 | 完成知识抽取、清洗、融合、本体映射，将原始数据转化为结构化/半结构化知识 |

| 知识服务层 | 封装检索、图谱查询、运营治理等原子能力，供上层调用，不直接面向最终用户 |
| 模型与智能编排层 | 承载大模型推理、RAG、Agent编排、Prompt与模型版本管理 |
| 应用层 | 面向最终用户的交互界面与开放API |

## 4. 分层架构详细设计

### 4.1 数据接入层

**核心组件：**

- **文档采集服务**：支持定时/触发式扫描指定目录、文档管理系统、邮件附件等来源，完成文档拉取；
- **数据库同步服务**：基于CDC（变更数据捕获）或定时批量同步方式，接入历史故障台账、传感器数据库；
- **专家录入前端**：结构化表单+富文本编辑器，支持图片、图表附件上传；
- **外部标准接入服务**：对接标准规范数据库（如已订阅），支持标准条款结构化抓取（需合规审查）。

**设计要点：**

- 所有接入数据统一封装为标准化"知识候选对象"（Knowledge Candidate），包含原始内容、来源元数据、密级标签；
- 接入层与加工层之间通过消息队列（如Kafka/RabbitMQ）解耦，支持削峰填谷与失败重试。

### 4.2 知识加工层

**核心组件：**

- **文档解析/OCR引擎**：支持PDF、Word、扫描件、图纸的文本与表格抽取；
- **知识抽取服务**：基于大模型+规则的NER（命名实体识别）、关系抽取、事件抽取子服务；
- **数据清洗与标准化服务**：术语归一化（同义词映射表）、格式校验、去重；
- **知识融合服务**：多源同类知识的融合、置信度加权、冲突检测；
- **本体映射服务**：将抽取结果映射至预定义本体的实体/关系类型，生成图谱实例化数据。

**处理流程（知识抽取主流程）：**

```
原始文档/数据 → 格式解析 → 文本分块(Chunking)
   → LLM辅助抽取(实体/关系/事件) → 规则校验与置信度评分
   → 术语标准化 → 人工复核队列(低置信度/新增术语强制复核)
   → 本体映射 → 生成知识候选条目 → 写入待审核知识库
```

**设计要点：**

- 抽取服务需支持**批处理**（历史存量数据迁移）与**流式处理**（新增文档实时处理）两种模式；
- 低置信度或与已有知识冲突的抽取结果，强制进入人工审核队列，不直接自动入库。

### 4.3 知识存储层

| 存储引擎 | 存储内容 | 选型方向 |
|---|---|---|
| 图数据库 | 本体、知识图谱实例（实体、关系、故障传播链） | 属性图模型，支持多跳查询与图算法 |
| 向量数据库 | 文档/知识条目Embedding，用于语义检索 | 支持十亿级向量、低延迟ANN检索 |
| 关系型数据库 | 用户权限、知识元数据、审核流程状态、结构化规则表、参数库 | 成熟关系型数据库，事务保证 |
| 时序数据库 | 传感器监测历史数据（如需知识库直接关联原始监测曲线） | 面向高频写入、区间查询优化 |
| 文档/对象存储 | 原始文档、图片、报告等非结构化文件 | 分布式对象存储 |
| 缓存层 | 高频检索结果、会话上下文缓存 | 内存缓存，支撑低延迟响应 |

**设计要点：**

- 四类存储引擎通过**知识服务层的统一数据访问接口（DAO/Repository）**对上屏蔽底层差异；
- 图数据库与向量数据库之间通过**知识条目唯一ID**关联，支持"语义检索命中→图谱关联扩展"的联合查询模式。

### 4.4 模型与智能编排层

**核心组件：**

- **基础大模型推理服务**：私有化部署的通用大模型推理引擎，提供统一推理API；
- **领域微调模型服务**：基于LoRA等参数高效微调技术产出的领域适配模型，支持按领域/任务动态加载；
- **RAG检索增强服务**：编排"查询理解→混合检索（向量+图谱+关键词）→上下文构建→模型生成→引用溯源"全流程；
- **Agent编排引擎**：基于任务规划（Planning）+工具调用（Tool Use）+反思（Reflection）机制，支撑多步骤复杂任务；
- **Prompt管理服务**：集中管理各业务场景Prompt模板，支持版本管理与A/B测试；
- **模型评估与灰度发布服务**：支持新模型/新Prompt上线前的离线评估与灰度验证。

**RAG服务详细流程：**

```
用户问题 → 查询改写/意图识别 → 权限与密级过滤条件生成
   → 并行检索：[向量语义检索] + [知识图谱关系检索] + [结构化规则检索]
   → 检索结果融合与重排序(Rerank)
   → 上下文拼装(含来源标注) → 大模型生成回答
   → 回答后处理(敏感信息过滤/引用标注) → 返回前端
```

**Agent编排设计要点：**

- Agent任务须定义明确的**工具集（Tool Registry）**，如：图谱查询工具、结构化数据库查询工具、检索工具、计算工具、外部PHM算法调用工具；
- 涉及安全关键判断的Agent任务，须在关键节点插入**人工确认节点**，不支持全自动闭环执行；
- Agent执行轨迹（每一步的思考、工具调用、结果）需完整记录，支撑可解释性与审计。

### 4.5 知识服务层

- **知识检索服务**：封装混合检索能力，供RAG服务与前端检索页面调用；
- **图谱查询服务**：提供基于Cypher/Gremlin等图查询语言的封装接口，支持多跳查询、路径查询、子图查询；
- **知识运营服务**：知识审核工作流引擎、版本管理、质量评估规则引擎；
- **统计分析服务**：知识资产统计、使用效果统计、专家贡献度统计。

### 4.6 应用层

| 应用模块 | 说明 |
|---|---|
| 智能问答Web端 | 面向全员的自然语言问答交互界面 |
| 知识检索门户 | 多维度检索、筛选、相似案例推荐 |
| 知识图谱可视化 | 交互式图谱探索工具 |
| 知识运营后台 | 面向专家/管理员的知识贡献、审核、统计管理界面 |
| 开放API网关 | 面向业务系统集成的统一API入口，含鉴权、限流、日志 |

## 5. 关键技术架构方案

### 5.1 混合检索架构（Hybrid Retrieval）

采用"向量检索+图谱检索+关键词/结构化检索"三路召回、统一重排序的架构，解决单一检索方式在专业领域的召回不足问题：

- 向量检索：解决语义相似但表述不同的召回（如"轴承异响"vs"轴承振动异常"）；
- 图谱检索：解决基于明确关系路径的精确召回（如"某型号发动机的历史故障模式"）；
- 关键词/结构化检索：解决型号、参数编号等精确匹配场景。

三路召回结果经过统一的Rerank模型（可基于轻量级模型或规则加权）融合排序后输入生成环节。

### 5.2 领域大模型能力增强架构

采用"基础大模型 + RAG + 领域微调"三级能力增强策略：

1. **基础大模型**：承担通用语言理解与生成能力；
2. **RAG检索增强**：解决知识时效性与专业细节准确性问题，无需重新训练即可更新知识；
3. **领域微调（SFT/LoRA）**：解决领域专业表达习惯、任务范式（如故障报告固定格式）的适配问题。

三级能力可根据实际效果评估逐步叠加投入，建议**优先建设RAG能力**（成本低、迭代快、可解释性强），微调作为后续能力增强手段。

### 5.3 知识图谱本体分层架构

```
核心本体层（Core Ontology）：装备-系统-部件层级、故障模式、征兆、诊断方法、
                            健康状态、寿命模型、维修策略（跨领域通用）
        │
        ├── 能源领域扩展本体（发电机组、风电、燃气轮机专有概念）
        ├── 交通领域扩展本体（轨道车辆、船舶专有概念）
        └── 航空航天领域扩展本体（航空发动机、机载系统、运载器专有概念）
```

该分层设计保证核心通用能力的复用，同时支持各领域专有知识的独立演进与扩展，新增业务领域时仅需扩展领域层，无需重构核心本体。

### 5.4 人机协同闭环架构

```
知识生产：AI初步抽取 → 专家审核确认 → 知识入库
知识应用：专家/工程师使用问答与检索 → 反馈满意度/纠错
知识反哺：反馈数据 → 知识更新/模型评估迭代 → 知识库与模型持续优化
```

该闭环通过知识运营服务层的反馈采集与统计分析组件支撑，形成知识资产与模型能力的持续演进机制。

## 7. 安全架构

### 7.1 安全分层设计

```
接入安全：统一身份认证(SSO/LDAP) + 终端准入控制
传输安全：内网/专网传输加密(TLS)
应用安全：接口鉴权(OAuth2/JWT) + 细粒度权限控制(RBAC+密级ABAC)
数据安全：分级分类标签 + 存储加密 + 脱敏处理
模型安全：私有化部署 + 输出内容安全审核 + 禁止敏感数据外呼公网API
审计安全：全链路操作日志 + 安全审计中心 + 异常行为告警
```

### 7.2 权限控制模型

采用**RBAC（基于角色的访问控制）+ ABAC（基于属性的访问控制，结合密级/领域属性）**混合模型：

- RBAC控制用户可执行的操作类型（查看/贡献/审核/管理）；
- ABAC结合知识密级、用户所属领域权限、项目授权范围，动态过滤检索与问答可访问的知识范围。

### 7.3 涉密数据处理原则

- 涉密数据处理环境与非涉密环境**物理/网络隔离**；
- 涉密知识不得进入面向非授权范围的向量库/模型训练语料；
- 模型推理服务部署于隔离环境内，不与公网存在数据交互通道。

## 8. 部署架构

### 8.1 部署拓扑（示意）

```
┌─────────────────────────────────────────────┐
│              公司内网/专网边界                    │
│  ┌───────────────┐   ┌───────────────────┐   │
│  │  应用服务集群    │   │  模型推理服务集群(GPU) │   │
│  │ (K8s容器化部署)  │   │  (GPU节点池)         │   │
│  └───────────────┘   └───────────────────┘   │
│  ┌───────────────┐   ┌───────────────────┐   │
│  │  知识存储集群    │   │  数据接入与加工集群    │   │
│  │(图库/向量库/关系库)│   │ (异步任务处理节点)    │   │
│  └───────────────┘   └───────────────────┘   │
│  ┌───────────────────────────────────────┐   │
│  │  安全与治理中心(认证/审计/监控/配置)         │   │
│  └───────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
        （涉密业务：独立隔离子环境，单独部署一套完整链路）
```

### 8.2 部署原则

- 采用容器化（Kubernetes）部署应用与知识服务层，支持弹性伸缩；
- GPU推理集群与其他服务集群资源隔离，避免资源争抢；
- 涉密业务领域建议**独立部署一套完整环境**，与非涉密环境物理隔离，不共享存储与模型服务。

---

## 9. 高可用与容灾架构

- 核心服务（应用层、知识服务层）建议采用多副本部署，配合负载均衡实现高可用；
- 知识存储层（图库/向量库/关系库）建议采用主从/集群模式，定期快照备份；
- 模型服务建议保留至少一个备用推理节点，支持故障自动切换（视预算分期实现）；
- 制定数据备份策略（全量+增量备份）与灾难恢复演练计划。

## 10. 技术选型建议

> 注：以下为技术方向性建议，具体产品需经过技术选型评审（安全合规性、性能测试、社区支持、私有化部署支持度等维度）后确定，本文档不做最终选型背书。

| 技术领域 | 选型方向建议 |
|---|---|
| 图数据库 | 具备成熟属性图模型、支持私有化部署、社区/商业支持完善的图数据库产品 |
| 向量数据库 | 支持十亿级向量规模、私有化部署、多种ANN索引算法的向量数据库产品 |
| 基础大模型 | 优先选择支持私有化部署、开源可控、具备中文及专业领域适配能力的模型 |
| 微调框架 | 支持LoRA等参数高效微调的成熟训练框架 |
| 容器编排 | Kubernetes |
| 消息队列 | 支持高吞吐、可靠投递的消息中间件 |
| 对象存储 | 支持私有化部署的分布式对象存储 |

## 11. 架构演进规划

| 阶段 | 架构建设重点 |
|---|---|
| 第一阶段（试点） | 建成数据接入-知识加工-知识存储-基础RAG问答的最小闭环，验证核心架构可行性 |
| 第二阶段（能力扩展） | 引入Agent编排、领域微调模型、图谱推理能力，完善知识运营闭环 |
