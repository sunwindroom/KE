## 1. 引言

### 1.1 编写目的

本需求规格说明书旨在明确公司内部"装备健康管理（PHM）领域知识工程与智能知识库系统"（以下简称"本系统"）的建设目标、功能范围、性能指标与约束条件，作为系统设计、开发、测试、验收及后续运维的依据，供项目组、研发团队、测试团队及各业务领域（能源、交通、航空、航天）专家评审使用。

### 1.2 项目背景

公司长期深耕装备健康管理（Prognostics and Health Management, PHM）系统研发与个性化定制软件集成，业务覆盖能源装备（发电机组、风电、燃气轮机等）、交通装备（轨道交通、船舶等）、航空航天装备（发动机、机载系统、运载器等）多个高价值、高可靠性要求领域。在长期项目交付过程中，公司积累了大量故障诊断案例、维修保障经验、传感器监测数据、技术标准规范、专家诊断经验等异构知识资产，但目前普遍存在以下问题：

- 知识分散在文档、报告、专家个人经验、历史系统数据库中，未形成统一、可复用的知识资产；
- 新项目、新型号的PHM系统建设仍高度依赖专家个人经验，知识复用效率低、周期长；
- 跨领域（能源/交通/航空航天）知识难以关联、迁移与类比复用；
- 缺乏面向PHM专业场景的智能问答、辅助诊断、辅助设计工具，专家经验难以规模化复用；
- 传统知识库系统（文档管理、检索）智能化程度低，无法支撑大模型时代的知识生产与应用范式。

为此，公司拟结合当前先进的大语言模型（LLM）、知识图谱、检索增强生成（RAG）、智能体（Agent）等AI技术，建设面向PHM领域的知识工程与智能知识库系统，实现知识的**获取—加工—存储—融合—应用—运营**全生命周期管理，支撑公司PHM产品研发、项目实施、专家经验传承与业务智能化升级。

## 2. 项目概述

### 2.1 项目名称

PHM领域知识工程与智能知识库系统建设项目

### 2.2 建设范围

本项目建设内容包括但不限于：

1. **知识中台**：面向PHM领域的知识获取、抽取、建模、存储、融合与治理平台；
2. **大模型能力底座**：私有化部署的基础大模型及领域增强能力（RAG检索、微调、Agent框架）；
3. **知识应用层**：智能问答、辅助诊断、知识检索、知识图谱可视化、报告生成、经验推荐等应用工具；
4. **知识运营体系**：知识贡献、评审、更新、质量评估、权限与安全管理的制度与系统支撑；
5. **与现有PHM业务系统的集成**：故障诊断系统、健康评估系统、寿命预测系统、维修决策系统等的知识服务接口。

### 2.3 项目干系人

| 角色 | 职责 |
|---|---|
| 公司管理层 | 项目立项、资源保障、战略决策 |
| PHM领域专家（能源/交通/航空航天） | 知识贡献、知识评审、业务场景验证 |
| 知识工程团队 | 知识建模、本体设计、知识治理 |
| AI/大模型研发团队 | 模型选型、微调、RAG/Agent开发 |
| 软件研发团队 | 系统开发、集成、测试 |
| IT运维团队 | 基础设施部署、安全运维 |
| 一线工程师/售前/交付人员 | 系统使用者，知识消费与反馈 |

### 2.4 建设原则

- **专业性优先**：以PHM领域知识体系为核心，AI能力服务于领域知识的生产与应用，而非替代专业判断；
- **安全可控**：涉及航空航天、能源等敏感领域数据，须满足私有化部署、数据不出域、分级分类管控要求；
- **人机协同**：强调专家经验与大模型能力的协同增强，而非完全自动化替代专家；
- **渐进迭代**：分领域、分阶段建设（可优先选择1-2个业务领域试点，逐步扩展至能源、交通、航空航天全领域）；
- **标准先行**：知识建模需遵循统一本体规范与元数据标准，保障跨项目、跨领域知识可复用。

## 3. 总体需求与建设目标

### 3.1 总体目标

建设一套覆盖"知识获取—知识加工—知识存储—知识融合—知识应用—知识运营"全链路的智能知识工程平台，实现：

1. 将分散在文档、系统、专家头脑中的PHM隐性知识与显性知识**结构化、资产化**；
2. 构建面向能源、交通、航空航天多领域的**PHM领域知识图谱**，支撑故障关联分析、根因追溯、类比推理；
3. 基于大模型构建**领域智能问答与辅助决策助手**，赋能诊断分析、方案设计、报告撰写、技术答疑等场景；
4. 建立**知识运营闭环机制**，使知识库能够持续积累、迭代、更新，避免"建成即过时"；
5. 为公司PHM产品与项目交付提供**可复用、可追溯、可度量**的知识服务能力，缩短新项目知识准备周期，提升专家经验复用率。

### 3.2 具体建设指标（示例，供评审阶段量化）

| 指标类别 | 指标说明 | 目标值（可根据实际调整） |
|---|---|---|
| 知识覆盖度 | 覆盖公司历史项目故障案例入库率 | 首期≥80%存量案例完成结构化入库 |
| 知识图谱规模 | 领域实体、关系数量 | 首期≥5万实体、≥20万关系（三领域合计） |
| 问答准确率 | 领域知识问答准确率（人工评估） | ≥85%（首期试点领域） |
| 检索响应 | RAG检索+生成响应时延 | ≤5秒（95分位） |
| 知识更新周期 | 新知识从提交到审核入库周期 | ≤3个工作日 |
| 专家复用率 | 知识条目被跨项目引用次数占比 | 逐年提升，年度可跟踪 |

### 3.3 用户角色与主要使用场景

| 角色 | 典型场景 |
|---|---|
| 领域专家 | 贡献故障案例、审核知识条目、维护本体规则 |
| 诊断/算法工程师 | 检索历史相似故障、辅助设计诊断规则与特征工程 |
| 项目交付工程师 | 快速检索技术方案、维修手册知识点、类比案例 |
| 售前/技术支持 | 快速生成技术问答、方案要点、竞品/标准对比 |
| 系统管理员 | 权限管理、系统监控、模型与知识库运维 |
| 管理层 | 知识资产统计、知识运营效果分析看板 |

## 4. 系统总体架构

### 4.1 总体架构分层

系统建议采用"五层"架构设计：

```
┌─────────────────────────────────────────────┐
│  应用层：智能问答/辅助诊断/知识检索/图谱可视化/  │
│         报告生成/知识运营门户/开放API         │
├─────────────────────────────────────────────┤
│  模型与推理层：基础大模型 + 领域微调模型 +      │
│      RAG检索增强 + Agent智能体编排 + 规则引擎   │
├─────────────────────────────────────────────┤
│  知识层：知识图谱（Ontology+实例）+向量知识库 + │
│      结构化知识库（规则/案例/参数） + 文档知识库 │
├─────────────────────────────────────────────┤
│  知识加工层：知识抽取/清洗/融合/质检/版本管理    │
├─────────────────────────────────────────────┤
│  数据接入层：文档、数据库、传感器/监测系统、     │
│      专家录入、外部标准/文献接入                │
└─────────────────────────────────────────────┘
        ↕（横向支撑：安全与权限、日志与监控、运维管理）
```

### 4.2 核心技术组件建议

| 组件 | 说明 |
|---|---|
| 文档解析与OCR引擎 | 处理PDF、Word、扫描件、图纸、传感器数据文件等多模态输入 |
| 知识抽取引擎（NER/关系抽取） | 基于LLM+规则的实体、关系、事件抽取 |
| 本体与图数据库 | 存储PHM领域本体及知识图谱实例（如Neo4j、NebulaGraph等，具体选型另行技术评审） |
| 向量数据库 | 存储文档/知识条目Embedding，支撑语义检索（如Milvus、Weaviate等） |
| 结构化知识库（关系型/时序库） | 存储诊断规则、故障模式库、参数库、寿命模型等结构化知识 |
| 基础大模型 | 私有化部署的通用大模型底座（支持SFT/LoRA微调） |
| RAG检索增强框架 | 结合向量检索+知识图谱检索+结构化检索的混合检索增强生成 |
| Agent编排框架 | 支持多步骤任务规划、工具调用（检索、计算、图谱查询、外部系统调用） |
| 知识运营与治理平台 | 知识贡献、评审、版本管理、质量评估、权限管理 |

### 4.3 架构设计约束

- 大模型部署须支持**私有化/本地化部署**，禁止涉密及敏感数据通过公网调用第三方模型API；
- 知识图谱本体设计需支持**多领域扩展**（能源/交通/航空航天领域本体可共享通用层、独立扩展专有层）；
- 系统需具备**模型可插拔**能力，支持未来更换/升级基础大模型而不影响上层应用与已有知识资产。

## 5. 功能需求

### 5.1 知识获取与知识抽取模块

#### 5.1.1 多源异构数据接入

- 支持接入结构化数据（关系型数据库、时序数据库中的传感器监测数据、历史故障台账）；
- 支持接入非结构化文档（Word、PDF、PPT、Excel、扫描件、图纸、CAD文件附带说明）；
- 支持接入半结构化数据（维修工单、故障报告表单、标准规范条款）；
- 支持专家在线知识录入（结构化表单+自由文本+图片/图表上传）；
- 支持批量导入历史项目知识资产（案例库、经验库迁移）。

#### 5.1.2 智能知识抽取

- 基于大模型的**实体识别**：识别装备、部件、故障模式、传感器参数、维修动作、时间、地点等领域实体；
- **关系抽取**：识别"部件-故障模式""故障-征兆""故障-根因""故障-处置措施"等关系；
- **事件与因果链抽取**：从故障报告、维修记录中抽取故障发展的时序因果链条；
- **知识蒸馏**：辅助专家将隐性经验（如"听声音判断轴承磨损"）转化为结构化规则条目；
- 抽取结果需支持**人工复核与修正**，形成"AI初抽+专家确认"的知识生产流程。

#### 5.1.3 数据清洗与标准化

- 术语标准化（同义词/别名归一，如"轴承""滚动轴承"统一映射）；
- 数据去重与冲突检测（同一故障案例多来源录入的合并与冲突提示）；
- 数据质量校验规则配置（必填项、格式校验、值域校验）。

### 5.2 知识建模与本体工程模块

#### 5.2.1 PHM领域本体设计

- 构建覆盖PHM通用概念的**核心本体层**，包括但不限于：装备/系统/部件层级结构、故障模式、故障机理、征兆/特征参数、诊断方法、健康状态、寿命预测模型、维修策略、维修资源；
- 构建**领域扩展本体层**，分别面向能源（发电机组、风电机组等）、交通（轨道车辆、船舶等）、航空航天（航空发动机、机载设备、运载器等）领域的专有概念与关系；
- 支持本体的**可视化编辑、版本管理与变更影响分析**。

#### 5.2.2 知识图谱构建

- 基于本体自动/半自动构建知识图谱实例层（装备台账、故障案例、诊断规则实例化为图谱节点与边）；
- 支持**跨领域知识关联**：如能源与交通领域共性部件（轴承、齿轮箱、电机）的故障模式类比与知识迁移；
- 支持**图谱质量评估**（完整性、一致性、准确性指标）与可视化探查工具。

#### 5.2.3 知识分类与元数据体系

- 建立统一知识分类体系（按领域、装备类型、知识类型：案例/规则/标准/文献/专家经验等多维度分类）；
- 定义知识元数据标准（来源、密级、置信度、有效期、责任专家、版本号等）。

### 5.3 知识存储与管理模块

- 支持知识图谱存储（图数据库）、向量化语义存储（向量数据库）、结构化规则/参数存储（关系型/时序数据库）、原始文档存储（文档管理系统）的统一管理与协同检索；
- 支持知识条目的**版本管理**（历史版本追溯、变更对比、回滚）；
- 支持知识的**生命周期管理**（草稿—待审—发布—失效—归档）；
- 支持大规模数据的存储扩展性设计，满足未来知识资产持续增长需求。

### 5.4 知识融合与知识推理模块

- **多源知识融合**：对来自不同项目、不同专家、不同文档的同类知识进行融合去重、置信度加权；
- **知识推理**：基于知识图谱的规则推理（如故障传播路径推理）与基于大模型的语义推理（相似案例类比、根因假设生成）相结合；
- **冲突检测与仲裁**：当不同来源知识存在矛盾（如不同专家对同一故障根因判断不一致）时，系统需提示冲突并支持专家仲裁流程。

### 5.5 大模型能力集成模块

#### 5.5.1 检索增强生成（RAG）

- 支持基于向量检索、图谱检索、关键词检索的**混合检索策略**，提升召回准确率；
- 支持检索结果的**引用溯源**（问答结果需标注知识来源，支持点击追溯原始文档/案例）；
- 支持检索范围按**领域、密级、项目**等维度进行权限过滤。

#### 5.5.2 领域大模型微调

- 基于公司PHM领域语料（历史报告、案例、标准）对基础大模型进行**领域适配微调**（全量微调/LoRA等参数高效微调方式，具体由技术团队评估）；
- 支持**多领域微调模型的管理与切换**（如能源领域问答模型、航空领域问答模型可分别或统一维护）；
- 建立**模型效果评估与迭代机制**，定期基于业务反馈数据回流优化模型。

#### 5.5.3 智能体（Agent）能力

- 支持构建面向典型PHM任务的**智能体工作流**，例如：
  - 故障诊断辅助Agent：接收故障现象描述，自动检索历史相似案例、关联知识图谱、给出可能根因排序及建议处置措施；
  - 寿命预测知识助手Agent：辅助工程师检索适用的寿命预测模型、参数取值范围及历史应用案例；
  - 报告撰写Agent：基于结构化诊断结果与知识库内容，辅助生成故障分析报告、维修建议报告初稿；
  - 标准/规范问答Agent：面向行业标准、规范条款的精准问答与合规性核查辅助。
- Agent需支持**工具调用**（图谱查询、结构化数据库查询、外部PHM算法接口调用、计算工具调用）；
- Agent执行过程需**可解释、可追溯**，关键决策环节保留人工确认节点（尤其涉及安全关键决策场景，如航空发动机健康判断，AI仅作辅助建议，不做自动裁决）。

### 5.6 知识应用与交互模块

#### 5.6.1 智能问答

- 支持自然语言问答，覆盖故障机理咨询、维修方案查询、标准规范解读、历史案例检索等场景；
- 支持多轮对话与上下文理解；
- 支持问答结果的**满意度反馈**与**纠错提交**，反馈数据用于知识库与模型优化。

#### 5.6.2 智能检索

- 支持关键词检索、语义检索、图谱检索多种检索方式；
- 支持按领域、装备型号、故障类型、时间范围等多维度筛选；
- 支持相似案例推荐（"猜你想找"）。

#### 5.6.3 知识图谱可视化

- 支持知识图谱的交互式可视化展示（节点展开、关系路径追溯、故障传播链可视化）；
- 支持面向不同角色的可视化视图（专家视图/管理视图/业务视图）。

#### 5.6.4 辅助报告与文档生成

- 支持基于知识库内容与检索结果，辅助生成技术方案初稿、故障分析报告初稿、知识摘要；
- 生成内容需支持人工编辑、审核与导出（Word/PDF等格式）。

#### 5.6.5 知识贡献与协同

- 支持专家在线贡献知识、评论、点评已有知识条目质量；
- 支持知识众包式积累（一线工程师可提交案例，经审核后入库）；
- 支持知识贡献激励与统计（贡献积分、贡献榜）。

### 5.7 知识运营与治理模块

- **知识审核流程**：新增/修改知识需经过责任专家审核后方可发布，支持多级审核配置；
- **知识质量评估**：定期对知识条目的准确性、时效性、完整性进行评估，标识过期或低质量知识；
- **知识更新提醒**：对关联标准变更、装备型号升级等触发相关知识条目的复核提醒；
- **知识资产统计与运营看板**：知识总量、增长趋势、覆盖领域分布、问答调用量、专家贡献度等运营指标可视化展示；
- **知识使用效果追踪**：记录知识在实际项目/诊断中的引用与效果反馈，形成知识价值闭环评估。

### 5.8 与公司现有PHM业务系统集成

- 提供标准化API/服务接口，支撑故障诊断系统、健康评估系统、寿命预测系统、维修决策支持系统等业务系统按需调用知识服务（如查询故障模式库、调用诊断规则、获取相似案例）；
- 支持将业务系统运行中产生的诊断结果、报警数据、专家复核结论**反哺**知识库，形成"业务应用—知识沉淀"闭环；
- 预留与公司现有项目管理系统、文档管理系统（如PLM/PDM）的数据对接接口。

### 5.9 系统管理模块

- 用户与权限管理（角色管理、领域权限、密级权限）；
- 组织架构与项目/领域知识空间管理（支持按业务领域划分知识空间，独立管理与共享设置）；
- 日志审计（操作日志、问答日志、知识变更日志留存与审计追溯）；
- 系统监控（模型服务状态、检索性能、资源占用监控与告警）；
- 模型与知识库版本管理、备份与恢复。

## 6. 数据需求

### 6.1 数据来源

| 数据类别 | 来源说明 |
|---|---|
| 历史项目文档 | 技术方案、验收报告、故障分析报告、维修手册等 |
| 装备监测数据 | 传感器时序数据、健康评估历史结果、报警记录 |
| 专家知识 | 专家访谈、经验录入、评审意见 |
| 标准规范 | 国家/行业/企业标准、适航/安全规范文档 |
| 外部公开资料 | 公开文献、技术论坛（需经合规审查后有限接入，不接入未经授权的付费/受限资源） |

### 6.2 数据分类分级

结合能源、交通、航空航天领域数据敏感性，需按照公司及行业保密管理要求，对知识条目进行**分级分类管理**（如：公开级、内部级、秘密级等），并在检索、问答、模型训练各环节严格执行分级访问控制，**涉密数据不得用于训练可能对外提供服务的模型，不得通过任何形式传输至公司内网/专网以外环境**。

### 6.3 数据质量要求

- 知识条目需具备可追溯来源；
- 结构化数据需符合预定义的数据字典与格式规范；
- 抽取生成的知识需标注置信度及是否经过人工审核。

## 7. 非功能需求

### 7.1 性能需求

| 指标 | 要求 |
|---|---|
| 知识检索响应时间 | 普通检索≤2秒；RAG问答生成≤5秒（95分位） |
| 并发支持 | 首期支持不少于200并发用户在线使用 |
| 知识图谱查询性能 | 三跳以内关系查询响应≤3秒 |
| 系统可用性 | 核心服务可用性≥99.5%（工作时间） |

### 7.2 可靠性与容灾

- 关键数据（知识图谱、向量库、结构化知识库）需定期备份，支持数据恢复；
- 系统需具备故障自动检测与告警能力；
- 建议核心组件具备高可用部署能力（视预算与规模分期实现）。

### 7.3 可扩展性

- 知识图谱本体、知识分类体系需支持后续新增业务领域（如新能源、船舶、轨道交通细分领域）的灵活扩展；
- 大模型底座需支持后续模型升级/替换，不影响已建知识资产。

### 7.4 易用性

- 提供面向不同角色（专家/工程师/管理者）的差异化交互界面；
- 提供知识贡献、审核操作的低门槛可视化工具，降低专家使用门槛；
- 支持中文为主的自然语言交互（如涉及国际化业务，可规划多语言扩展）。

### 7.5 可维护性

- 系统各模块（数据接入、知识加工、模型服务、应用层）需松耦合设计，便于独立升级维护；
- 提供完善的系统运维文档、操作手册及知识建模规范文档。

### 7.6 合规性

- 系统建设与数据处理须符合《中华人民共和国网络安全法》《数据安全法》《个人信息保护法》及军工/航空航天行业保密管理相关规定；
- 涉及军工/涉密项目知识资产，需按国家保密管理要求进行物理隔离部署与专门审批流程。

## 10. 安全保密要求

鉴于本系统服务于能源、交通、航空航天等对数据安全高度敏感的领域，安全保密要求为本项目**一票否决级**约束条件，具体包括：

1. **数据不出域**：所有涉及企业内部技术资料及敏感项目数据的处理、存储、模型训练/推理均须在公司可控环境内完成，禁止调用任何未经安全评估的公网大模型API处理敏感数据；
2. **分级分类访问控制**：基于知识密级、用户角色、所属领域实施细粒度的检索与问答权限控制，确保用户仅能访问其权限范围内的知识；
3. **模型输出安全**：对大模型生成内容需进行必要的内容安全审核，避免生成误导性、错误性的安全关键建议被直接用于决策而未经专家复核，尤其在航空、航天等安全关键领域，**系统输出仅作辅助参考，不替代专业人员最终判断与签发**；
4. **审计与追溯**：所有知识检索、问答、知识变更操作须留痕，支持安全审计；
5. **数据脱敏与最小化**：知识抽取、模型训练语料需进行必要的脱敏处理，避免过度暴露具体型号/项目敏感细节于非授权范围；
6. **供应商与开源组件安全评估**：所选用大模型、开源框架及第三方组件须经过安全评估，避免引入未知安全风险（如后门、数据回传等）。

### 附录C：风险提示

- **知识获取风险**：专家参与知识贡献与审核的时间投入可能成为进度瓶颈，建议配套激励机制；
- **模型幻觉风险**：大模型在专业领域可能产生看似合理实则错误的输出，须通过RAG溯源、专家复核机制加以控制，严禁在安全关键场景直接采信未经复核的模型输出；
- **数据安全风险**：多领域敏感数据汇聚后风险集中度提高，需强化分级分类与访问审计；
- **知识陈旧化风险**：装备型号、标准规范持续更新，需建立常态化知识更新机制，避免知识库"建成即过时"。
